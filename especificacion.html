DOCUMENTO DE ESPECIFICACIÓN

Plataforma de Administración y Realización de Evaluaciones

Sistema de cuestionarios dinámicos con lógica condicional, diseñado para usuarios no técnicos mediante una interfaz visual de tipo drag-and-drop. Orientado a encuestas, evaluaciones clínicas, escalas médicas y formularios de admisión complejos.

Versión 1.0 Junio 2026 Tipo: Plataforma SaaS

01Objetivo del sistema

Definición del propósito central de la plataforma.

El sistema debe permitir a usuarios no técnicos diseñar cuestionarios dinámicos mediante una interfaz visual de arrastrar y soltar, donde las preguntas se conectan entre sí mediante reglas de negocio y flujos condicionales — sin necesidad de escribir código ni depender de soporte técnico.

Inspiración funcional: combina la simplicidad de Typeform, la potencia lógica de SurveyMonkey Advanced Logic, y la flexibilidad visual de un motor BPM ligero (estilo Miro/Lucidchart).

03Motor de lógica condicional

Reglas visuales para construir flujos inteligentes entre preguntas.

El motor debe soportar operadores lógicos completos para construir condiciones simples o compuestas:

IF / THEN ELSE AND OR Reglas anidadas

Comportamiento esperado

  • Si la respuesta a una pregunta cumple una condición, se muestran o se saltan preguntas específicas.
  • Varias rutas pueden converger nuevamente en una misma pregunta posterior.
  • Una pregunta puede tener múltiples caminos de salida dependiendo de la opción seleccionada.
  • Las reglas pueden anidarse (condiciones dentro de condiciones) y combinarse con AND / OR.

Ejemplo de flujo

Ambas rutas (Sí / No) convergen nuevamente en la Pregunta 5, evitando duplicación de lógica.

11Stack tecnológico

Tecnologías definidas para la construcción y despliegue de la plataforma.

⚛️ Frontend — React

La aplicación (constructor visual, dashboard y vista de cuestionario final) se desarrolla en React, consumiendo una capa de API que se comunica con la base de datos Oracle.

🗄️ Backend — Oracle

La persistencia de datos (cuestionarios, preguntas, flujos, reglas, respuestas) se gestiona sobre Oracle Database, según el modelo relacional definido en este documento.

▲ Despliegue — Vercel

El frontend React se monta y despliega en Vercel. La conexión a Oracle se realiza mediante una capa de API/servicios intermedia, ya que Vercel no expone conexiones directas a bases de datos on-premise.

🔌 Integración

La integración entre el frontend y Oracle se realiza en su totalidad mediante un paquete PL/SQL llamado pkgln_cuestionarios. Todos los parámetros de entrada se envían como un JSON único en el parámetro p_json_entrada, y la respuesta se devuelve como un JSON único en el parámetro de salida p_json_salida.

Para la autenticación se utiliza la función pkgln_seguridad.f_validar_clave, a la que se le pasa el usuario, la clave y p_id_rol = 12 (rol Coordinador Médico). Si la función devuelve 1, el usuario tiene permiso para ingresar.

12Impresión de cuestionarios

Capacidad de generar una versión imprimible de cada cuestionario.

  • Los cuestionarios deben poder imprimirse, tanto en su versión vacía (para diligenciamiento en papel) como con las respuestas ya registradas.
  • La vista de impresión debe ocultar elementos de navegación, botones y controles interactivos, mostrando únicamente preguntas, opciones y resultados.
  • Debe respetar el orden visual definido por el flujo condicional realmente recorrido por el usuario (no todas las preguntas del cuestionario, sino las efectivamente mostradas).
  • Incluir encabezado con nombre del cuestionario, fecha y, cuando aplique, puntaje total y clasificación final.
  • Generación de versión imprimible en PDF como alternativa a la impresión directa del navegador.

13Modelo relacional

Estructura de base de datos Oracle — versión documentada con catálogo de tipos de pregunta, secciones de cuestionario y catálogo de operadores.

Diagrama de flujo de datos

Cuestionario → Secciones → Preguntas (tipificadas por catálogo) → Opciones / Asociaciones → Flujos y Reglas → Instancia de respuesta → Respuestas → Puntaje / Clasificación

Convenciones generales

  • Todas las tablas inician con el prefijo tkr_.
  • Llave primaria: columna id NUMBER.
  • Cada tabla tiene su secuencia asociada: <tabla>_seq.
  • Trigger BEFORE INSERT por tabla para asignar la PK automáticamente desde la secuencia.
  • Columna estado: 0 = Inactivo, 1 = Activo.
